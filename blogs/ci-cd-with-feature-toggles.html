Continuous testing in a CICD setup with feature flags!

Traditionally, the only way to "release" new features to customers is by "deploying" the source code of new feature to production. The release and deployment happens at the same time and are thus tightly coupled. A feature toggling tool such as LaunchDarkly, allows teams to decouple the release and deployment of new features. This means, the team can deploy the code of new feature to production, but not release it to customers. This decoupling brings in a lot of flexibility and benefits to the team, Such as, Having separate goals for deployment (done via CICD pipelines) and releases (done via LaunchDarkly).

Deployment Goals!

  1. Don't introduce new issues or vulnerabilites in our systems while bringing in new features.
    1. Formating checks should pass.
    2. Linting checks should pass.
    3. Security scans should not have any high or critical vulnerabilities <=7 rating.
    4. License checks should be compliant with open source standards.
    5. Integration tests should pass.
  2. Don't break or degrade existing functionality.
    1. All unit tests should pass.
    2. All integration tests should pass.
    3. All existing system/service tests should pass.
    4. Test coverage should not drop below a team threshold (of say 80%).

Release Goals!

  1. New functionality meets expectations and is ready to be released.
    1. Functional testing is successful.
    2. Exploratory testing is successful.
    3. Performance testing is successful.
    4. Business Acceptance testing is successful.

Potential benefits of decoupled release and deployments!

  1. Simple and risk free Trunk Based Development.
  2. Potential to deploy to production multiple times a day without accidentally releasing half-baked features to customers.
  3. Potential to release on demand, when ready.
  4. Potential to soft rollback (by hiding the faulty functionality from customer) on click of a button.
  5. No need to do hotfixes. Faults can be fixed carefully, without under pressure, and same as any other feature or bugfix.
  6. Simplified release process.
  7. Simplified and standardized deployment process for any and all kind of changes.
  8. Potential to do A/B testing and canary releases in production.
  9. No need for multiple test environments (dev, test, staging).
  10. Potentially one Test environment with targetting rules set in LaunchDarkly should be sufficient to cover all scenarios.
